二、架构设计

项目严格遵循 MVC 架构，将数据、视图、逻辑三者分离。各层通过回调接口通信，避免双向依赖。

分层职责

层级目录职责关键类
Config层 configs/ 静态配置，加载JSON关卡数据和资源路径 LevelConfig, LevelConfigLoader, CardResConfig
Model层 models/ 运行时数据模型，不含业务逻辑 CardModel, GameModel, UndoModel
View层 views/ UI展示 + 用户输入捕获，不含业务逻辑 CardView, GameView, PlayFieldView, TrayView, StackView
Controller层 controllers/ 协调Model与View，处理业务逻辑 GameController, PlayFieldController, StackController
Manager层 managers/ Controller的成员，管理Model数据加工，禁止单例 UndoManager
Service层 services/ 无状态业务逻辑，不持有数据 CardMatchService, GameModelFromLevelGenerator
Utils层 utils/ 通用工具函数 CardUtils

层间依赖关系

Controller  ---->  Model      (持有并修改)
Controller  ---->  View       (调用动画接口更新UI)
Controller  ---->  Manager    (作为成员变量)
Controller  ---->  Service    (调用静态方法)
View        ---->  Controller (通过回调接口，不直接引用)
Manager     ---->  Model      (持有并加工)
Manager  --X-->  Controller (禁止反向依赖)
Service  --X-->  Controller (不依赖Controller)
Service  --X-->  Model数据   (不持有，只加工)

四、核心交互流程

1. 游戏初始化

GameScene::init() → 创建背景层 → GameController::init() → LevelConfigLoader 加载JSON → GameModelFromLevelGenerator 生成Model → 创建GameView → 初始化PlayField/Stack/Tray → 注册回调

2. 桌面牌匹配（牌区 → 手牌区）

用户点击卡牌 → CardView触发回调 → PlayFieldController 验证状态 → CardMatchService 检查匹配 → GameModel 更新数据 → UndoManager 记录操作 → 卡牌重挂到GameView → EaseInOut动画飞向手牌区 → 更新TrayView + 翻开被遮盖的牌

3. 备用牌抽牌（牌堆 → 手牌区）

用户点击牌堆 → StackController 触发回调 → GameModel pop牌堆顶牌 → UndoManager 记录 → 创建动画卡牌于牌堆位置 → 移动+翻转动画飞向手牌区 → 更新TrayView和StackView计数

4. 回退操作

用户点击回退 → UndoManager pop记录 → GameModel 恢复数据 → 创建动画卡牌于手牌区位置 → EaseInOut动画飞回原位/牌堆 → 创建最终CardView + 恢复遮盖关系

五、Bug修复记录

以下是开发/调试过程中发现并修复的所有Bug：

Bug #0：编译错误 - MSVC UTF-8 编码问题

proj.win32/CardGame.vcxproj 原因 源码中含中文注释（UTF-8编码），MSVC默认按GBK解析导致语法错误（C2059, C4430等）
修复 在vcxproj的Debug和Release配置中添加 /utf-8 编译选项 
<AdditionalOptions>/utf-8 %(AdditionalOptions)</AdditionalOptions>

Bug #1：回退后崩溃 - CardView未注册到Map

GameController.cpp / PlayFieldView.h 原因 回退操作恢复卡牌时，新创建的CardView只添加到了场景树（addChild），没有注册到PlayFieldView的 _cardViews map中。后续对该卡牌的任何操作（flipCard、getCardView）都找不到它，导致空指针崩溃。
修复 新增 PlayFieldView::addCardView(int, CardView*) 方法，在回退时同步注册到map 
// 回退时注册视图
_gameView->getPlayFieldView()->addChild(restoredView);
_gameView->getPlayFieldView()->addCardView(card->getId(), restoredView); // 新增

Bug #2：flipCard始终显示正面

PlayFieldView.cpp 原因 翻牌动画的回调中硬编码调用 showFront()，没有检查CardModel的实际faceUp状态。导致需要翻回背面的牌也显示正面。
修复 修改为根据 getCardModel()->isFaceUp() 决定显示正面还是背面 
cardView->playFlipAnimation([cardView, shouldBeClickable]() {
    if (cardView->getCardModel() && cardView->getCardModel()->isFaceUp())
        cardView->showFront();
    else
        cardView->showBack();
    cardView->setClickable(shouldBeClickable);
});

Bug #3：翻开的牌不可点击

PlayFieldView.cpp 原因 flipCard() 只播放翻转动画，但没有更新卡牌的clickable状态。被解除遮盖并翻开的牌无法被点击。
修复 在翻转动画回调中根据 isFaceUp() && !isCovered() 设置 setClickable()

Bug #4：遮盖关系重复添加

CardModel.cpp 原因 addCoveredBy() 没有去重检查。回退操作反复执行时，同一个遮盖ID被重复添加到 _coveredByIds 列表中，导致 removeCoveredBy() 一次只能移除一个，遮盖关系无法正确解除。
修复 添加 std::find 去重判断 
void CardModel::addCoveredBy(int id) {
    auto it = std::find(_coveredByIds.begin(), _coveredByIds.end(), id);
    if (it == _coveredByIds.end()) {
        _coveredByIds.push_back(id);
    }
}

Bug #5：牌区→手牌区动画坐标错误（卡牌飞到屏幕外）

GameController.cpp 原因 原代码使用 TrayView::getTopCardPosition()（返回的是GameView空间坐标）传入 convertToWorldSpace()（期望TrayView本地坐标），导致坐标转换错误，目标位置偏离屏幕。
修复 改为将CardView从PlayFieldView重挂到GameView上，直接在GameView坐标系中计算目标位置： 
// 获取卡牌的世界坐标，转换到GameView空间
Vec2 worldPos = cardView->getParent()->convertToWorldSpace(cardView->getPosition());
Vec2 gameViewPos = _gameView->convertToNodeSpace(worldPos);
// 重挂到GameView
cardView->retain();
_gameView->getPlayFieldView()->detachCardView(cardId);
cardView->removeFromParent();
cardView->setPosition(gameViewPos);
_gameView->addChild(cardView, 50);
cardView->release();
// 目标：手牌区中心
Vec2 trayPos = _gameView->getTrayView()->getPosition();
Vec2 trayTarget(trayPos.x - cardSize.width*0.5f, trayPos.y - cardSize.height*0.5f);

Bug #6：回退时CardView创建后model已改状态

GameController.cpp 原因 回退牌堆→手牌区时，先设 card->setFaceUp(false) 再创建CardView，CardView的init读到faceUp=false直接显示背面，后续手动showFront无效。
修复 调整顺序：先在faceUp=true时创建CardView，再修改model状态

七、动画系统详解

所有卡牌操作都通过 _isAnimating 标志锁定，动画期间禁止新的操作，防止状态混乱。

操作动画类型时长缓动技术要点
牌区 → 手牌区 MoveTo 0.3s EaseInOut CardView从PlayFieldView重挂到GameView，避免跨节点坐标转换问题
牌堆 → 手牌区 MoveTo + ScaleX翻转 0.25s EaseInOut 临时CardView在GameView上；翻转通过scaleX(1→0→1)+showFront实现
回退(牌区) MoveTo 0.3s EaseOut 临时CardView从手牌区飞到原位，到达后创建最终视图并注册到map
回退(牌堆) MoveTo + ScaleX翻转 0.25s EaseInOut 移动中途通过scaleX缩放切换为背面
翻牌 ScaleX(1→0→1) 0.3s 线性 scaleX缩到0时切换正/反面显示，再恢复

翻转动画原理

// CardView::playFlipAnimation - 通过scaleX缩放模拟3D翻牌
auto scaleDown = ScaleTo::create(0.15f, 0.0f, 1.0f);  // X缩到0
auto switchFace = CallFunc::create([this]() {
    if (_cardModel->isFaceUp()) showFront();
    else showBack();
});
auto scaleUp = ScaleTo::create(0.15f, 1.0f, 1.0f);    // X恢复
auto sequence = Sequence::create(scaleDown, switchFace, scaleUp, onComplete, nullptr);

跨区域动画原理

卡牌从牌区移到手牌区时，由于两者是不同Node的子节点，直接在PlayFieldView内做MoveTo无法正确到达TrayView的位置。解决方案：

// 1. 获取卡牌世界坐标
Vec2 worldPos = cardView->getParent()->convertToWorldSpace(cardView->getPosition());
Vec2 gameViewPos = _gameView->convertToNodeSpace(worldPos);

// 2. 从PlayFieldView取出，挂到GameView（公共父节点）
cardView->retain();
playFieldView->detachCardView(cardId);  // 只从map移除
cardView->removeFromParent();           // 从PlayFieldView移除
cardView->setPosition(gameViewPos);     // 设置GameView空间坐标
_gameView->addChild(cardView, 50);      // 挂到GameView
cardView->release();

// 3. 在GameView坐标系中计算目标位置并动画
Vec2 trayTarget(trayPos.x - cardSize.width*0.5f, trayPos.y - cardSize.height*0.5f);
cardView->runAction(EaseInOut::create(MoveTo::create(0.3f, trayTarget), 2.0f));

八、扩展指南（加分项）

如何新增一种卡牌类型

例如：添加"Joker"（大小王）。

  1. Model层：在 CardModel.h 的枚举中添加新值 
    enum CardFaceType {
    // ... 原有类型
    CFT_KING,
    CFT_JOKER,          // 新增
    CFT_NUM_CARD_FACE_TYPES
};
enum CardSuitType {
    // ... 原有类型
    CST_JOKER_RED,      // 新增（如需）
    CST_JOKER_BLACK,    // 新增
};
  2. Config层：在 CardResConfig.h 的路径方法中添加对应资源 
    // getBigNumberPath中添加
case CFT_JOKER: return "res/number/big_joker.png";
// getSuitPath中添加
case CST_JOKER_RED: return "res/suits/joker_red.png";
  3. Service层：在 CardMatchService 中更新匹配规则（如Joker可匹配任意牌） 
    static bool canMatch(CardFaceType f1, CardFaceType f2) {
    if (f1 == CFT_JOKER || f2 == CFT_JOKER) return true; // 万能牌
    return std::abs((int)f1 - (int)f2) == 1;
}
  4. 关卡JSON：在 level_1.json 的Playfield或Stack中添加新牌数据

不需要修改：View层、Controller层、Manager层 —— 它们都通过枚举和接口工作，自动适配新类型。

如何新增一种回退类型

例如：添加"特殊技能使用"的回退。

  1. Model层：在 UndoModel.h 的枚举中添加新类型 
    enum UndoActionType {
    UNDO_PLAYFIELD_TO_TRAY,
    UNDO_STACK_TO_TRAY,
    UNDO_SKILL_USE          // 新增
};
    如需额外数据，扩展 UndoRecord 结构体： 
    struct UndoRecord {
    UndoActionType type;
    int cardId;
    cocos2d::Vec2 originalPosition;
    std::vector<int> uncoveredCardIds;
    int skillType;              // 新增：技能类型
    std::string extraData;      // 新增：额外数据
};
  2. Manager层：在 UndoManager 中添加记录方法 
    void recordSkillUse(int cardId, int skillType) {
    UndoRecord record;
    record.type = UNDO_SKILL_USE;
    record.cardId = cardId;
    record.skillType = skillType;
    _undoModel->pushRecord(record);
}
  3. Controller层：在 GameController::onUndoClick() 中添加新分支 
    } else if (record.type == UNDO_SKILL_USE) {
    // 恢复技能使用前的状态
    // 创建动画...
}

设计要点：UndoRecord是纯数据结构，所有回退逻辑在Controller中处理。Manager只负责记录和弹出，不包含恢复逻辑。

如何新增一个关卡

  1. 在 Resources/levels/ 下创建新JSON文件（如 level_2.json）
  2. 按照现有格式定义Playfield和Stack数组
  3. 修改 GameController::init() 中的关卡文件路径，或实现关卡选择UI
